# 自建小智 vs 官网小智：验证码播报差异（问题可能在自建 server）

## 现象对比

| 场景 | 官网小智 server | 自建 xiaozhi-esp32-server |
|------|-----------------|---------------------------|
| 未绑定设备时，唤醒后 | **音箱会语音播报验证码** | 有时只看到日志有验证码，**音箱没有播报**或播报异常 |

你的感觉：问题更可能出在自建小智 server 的返回格式/顺序，而不是 Open-XiaoAI。

---

## Open-XiaoAI 端「能播报」的硬性要求

客户端只有在 **device_state == SPEAKING** 时才会把收到的**二进制音频**写入扬声器（即真正播出来）：

```text
_on_incoming_audio(data)  [xiaozhi.py]
  → 仅当 device_state == DeviceState.SPEAKING 时
  → self.audio_codec.write_audio(data)   // 这里才会出声
```

而 **SPEAKING** 只会在收到服务器发来的 **TTS 开始** 时被设置：

```text
_on_incoming_json → type == "tts" 且 state == "start"
  → _handle_tts_message(data)
  → _handle_tts_start() → set_device_state(DeviceState.SPEAKING)
```

因此，**无论官网还是自建，只要希望音箱播报验证码，server 端必须**：

1. **先发**一条 JSON：`{"type":"tts","state":"start", ...}`（可带 `session_id`），让客户端进入 SPEAKING。
2. **再发**二进制 Opus 音频帧（或客户端能识别的音频格式）。
3. 可选：发 `{"type":"tts","state":"sentence_start","text":"请登录控制面板，输入xxxxxx，绑定设备"}` 用于日志/界面。
4. 最后发 `{"type":"tts","state":"stop", ...}` 结束 TTS。

如果自建 server **没发 `tts` + `state: "start"`**，或**顺序错了**（例如先发音频再发 start），客户端就不会进入 SPEAKING，**所有后续音频都会被忽略**，音箱就不会播报验证码。

---

## 自建 server 可能的问题点

1. **没发 `type: "tts", state: "start"`**  
   只发了「第一段语音」的二进制或只发了 `sentence_start` 的 text，客户端不会切到 SPEAKING，自然不会播报。

2. **发送顺序不对**  
   例如先发二进制音频，再发 `tts` start；或先发 `sentence_start` 再发 start。客户端要求**先 start，再收音频**。

3. **消息格式与官网不一致**  
   - 官网：`type` / `state` 等字段名、取值和 Open-XiaoAI 里 `_handle_tts_message` 的预期一致。  
   - 自建：若用了不同字段名（如 `state` 写成 `status`）或不同取值，客户端不会识别为 TTS start，也不会进入 SPEAKING。

4. **连接过早关闭**  
   若在发完 `tts start` + 音频之前就断开连接，或 abort 后立刻断开，客户端也来不及播报。

---

## 建议在自建小智 server 侧做的核对

在 **xiaozhi-esp32-server** 里找到「向 WebSocket 客户端下发 TTS / 验证码」的代码（例如发送「请登录控制面板，输入 xxxxxx，绑定设备」和对应音频的模块），确认：

1. **是否在发任何 TTS 音频之前**，先发了一条 JSON：  
   `type == "tts"` 且 `state == "start"`（或等价字段）。
2. **发送顺序**是否为：  
   `tts start` → 二进制音频（可选中间有 `sentence_start` 文本）→ `tts stop`。
3. **字段名与取值**是否与 Open-XiaoAI 的 `_handle_tts_message` 一致（至少 `type`、`state`）。

若你使用 xiaozhi-esp32-server 的官方/开源版本，可对比其与**官网小智 server** 的 WebSocket 下行协议文档或抓包，看官网在「未绑定设备时下发验证码」时，第一条 JSON 和后续音频的顺序与格式。

---

## 如何验证「问题在自建 server」

1. **抓包或打日志**  
   在自建 server 向客户端发「验证码提示」的那段逻辑里，打印或抓包：  
   - 发出的第一条 JSON（是否包含 `type: "tts", state: "start"`）。  
   - 之后是否发了二进制音频、是否在 start 之后。

2. **临时用官网 server**  
   同一 Open-XiaoAI 配置改为连官网小智，未绑定设备时唤醒，确认能听到验证码播报，说明客户端逻辑正常。

3. **对比协议**  
   若有官网的 WebSocket 协议说明或示例（如 [小智 WebSocket 文档](https://xiaozhi.dev/docs/development/websocket/)），逐条对比自建 server 的下行消息格式和顺序。

## 自查结果（基于现有 patches）

本地只有小智 server 的**部分补丁**（`connection.py`、`ota_handler.py`、`websocket_server.py`），没有 TTS 下发到客户端的那一层代码，所以无法在这里直接看到「是否先发了 `tts` + `state: "start"`」。

从现有代码能确定的是：

- **验证码提示的触发**：`connection.py` 里在 `need_bind` 时会走 `_discard_message_with_bind_prompt()`，里面调用 `core.handle.receiveAudioHandle.check_bind_device(self)`，由那里去播「请登录控制面板，输入 xxxxxx，绑定设备」。
- **TTS 入队**：对话/绑定提示等内容是通过 `self.tts.tts_text_queue.put(TTSMessageDTO(...))` 交给 TTS 模块的；真正往 WebSocket **发** JSON/二进制的是 TTS 侧或 **`core.handle.sendAudioHandle`**（你 NAS 日志里 `[core.handle.sendAudioHandle]-INFO-发送第一段语音` 就是这一层）。
- **结论**：**是否先发 `type: "tts", state: "start"`** 必须在 **xiaozhi-esp32-server 源码**里查 **`core/handle/sendAudioHandle.py`**（或当前版本里等价「向 WebSocket 发 TTS」的模块）。若仓库里没有，需要从运行在 NAS 上的镜像/代码里找到该文件，确认发送顺序是否为：先一条 `{"type":"tts","state":"start"}`，再发二进制音频，再 `state: "stop"`。

## 「20:58:07 那条连接」是谁

小智 server 日志里写的 **device-id `02:42:c0:a8:90:02`** 是**连到小智的那一端**的设备标识，也就是：

- **谁连的**：**Open-XiaoAI**（你在 NAS 上跑的那个 Docker 服务 `open-xiaoai-xiaozhi`）。
- **这个 ID 从哪来**：Open-XiaoAI 连小智 WebSocket 时，会把当前环境的「设备 ID」放进请求头 `Device-Id`。在 Docker 里通常用**容器的 MAC 地址**作为设备 ID；`02:42:c0:a8:90:02` 就是那次重建/重启 Open-XiaoAI 容器后，该容器在 Docker 网络里的 MAC。
- **时间线**：20:58 左右你改成了「Docker 网络」配置并重建了 Open-XiaoAI 容器，所以新容器的 MAC 变成了 `02:42:c0:a8:90:02`，之后那次「你好小智」的连线和验证码 362476 就是这条连接。

所以：**20:58:07 那条连接 = 你 NAS 上的 Open-XiaoAI 容器，以设备 02:42:c0:a8:90:02 的身份连到自建小智**；不是音箱直接连小智，也不是别的服务。

## 小结

- **官网小智**：未绑定时也能语音播报验证码，说明其 TTS 下行顺序和格式符合 Open-XiaoAI 的预期（先 `tts` start，再音频）。
- **自建 server**：若**从未**或**未在正确时机**发送 `type: "tts", state: "start"`，或顺序/格式与客户端不一致，就会出现「日志有验证码、但音箱不播报」的现象。
- **自查**：本地只有 connection/ota/websocket 补丁，看不到 TTS 发送顺序；需在 **xiaozhi-esp32-server 源码**里查 **sendAudioHandle**（或等价模块），确认是否先发 `tts` + `state: "start"` 再发音频。
- 建议**优先在自建 xiaozhi-esp32-server 里核对并修正 TTS 下行的顺序与格式**，使其与官网行为一致，再观察验证码是否能在音箱正常播报。
