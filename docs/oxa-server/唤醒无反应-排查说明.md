# 已连接但唤醒词无反应 — 当前结论与建议

## 已确认正常的部分

- 音箱说「已连接」→ 说明 **Rust Client 已连上 NAS:4399**，Server 能下发指令。
- NAS 容器内 **keywords.txt** 含「小智同学」「你好小智」等，**config.py / VAD 配置** 已按文档和 Issue 调过（含 boost、低 threshold）。
- 部署方式：**Docker + CLI 模式**，与官方文档一致。

## 可能原因（需进一步验证）

1. **音箱端录音/上传**  
   社区有反馈：Client 用 arecord 等工具有录制上限，时间一长可能不再往 Server 送音频，导致 Server 端永远收不到语音，唤醒就不会触发。你这边是「一连接就完全唤不醒」，也可能是**麦克风音频根本没稳定送到 Server**（或格式/采样率不匹配）。

2. **Server 端未收到/未处理音频**  
   Docker 里进程没有把收到的音频正确交给 VAD/KWS，或只在「非 Docker」环境下测试过，在容器里存在兼容问题。

3. **设备/环境差异**  
   不同型号小爱、不同固件/Client 版本，表现可能不同；你当前型号 + NAS Docker 的组合可能尚未被充分验证。

## 建议你做的两件事

### 1. 在 Mac 上直接跑 Server（不经过 Docker）做对比

在 Mac 上克隆仓库，用**同一份 config** 跑官方「编译运行」方式，看同一只音箱、同一网络下能否唤醒：

```bash
git clone https://github.com/idootop/open-xiaoai.git
cd open-xiaoai/examples/xiaozhi
# 把你的 config.py 拷过来，或只改 keywords / vad
CLI=true uv run main.py
```

- 音箱临时把 `server.txt` 改成 **Mac 的局域网 IP:4399**，让音箱连 Mac 上的 Server。
- 若 **Mac 上能唤醒、NAS Docker 不能** → 多半是 Docker/网络/环境问题。
- 若 **两边都不能** → 更可能是音箱/Client 录音或上传有问题。

### 2. 到 GitHub 提 Issue，附上你的环境信息

到 [idootop/open-xiaoai Issues](https://github.com/idootop/open-xiaoai/issues) 开一个 issue，说明：

- 现象：**能听到「已连接」，但说「小智同学」「你好小智」「豆包豆包」等均无任何反应**。
- 环境：NAS Docker 部署（镜像 `docker.1ms.run/idootop/open-xiaoai-xiaozhi:latest`），小爱型号、固件版本，Rust Client 用 Gitee init.sh 安装、已设开机自启。
- 已做：config 里加了 vad boost、threshold 调低、keywords 含上述词，容器内 keywords.txt 已确认正确。

这样维护者或社区更容易判断是已知问题（如 arecord/音频流）还是需要额外排查。
