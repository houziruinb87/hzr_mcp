# 「爸爸最帅」/ 唤醒无响应、从未听到验证码 — 排查说明

## 1. 配置与关键词已就绪

- NAS 上的 `config_oxa.py` 已包含唤醒词 **「爸爸最帅」「你好小智」**，且容器内手动执行 `keywords.py` 会正确生成对应 `keywords.txt`。
- 若仍完全无响应，多半是 **音箱并未连到当前 NAS 的 Open-XiaoAI Server**，而不是配置文件错误。

## 2. 必须满足的前置条件（官方要求）

根据 [open-xiaoai xiaozhi 文档](https://github.com/idootop/open-xiaoai/tree/main/examples/xiaozhi)：

> 继续下面的操作之前，你需要**先在小爱音箱上启动运行 Rust 补丁程序**。

也就是说：

- 小爱音箱必须已刷/跑 **Open-XiaoAI 的 Rust 补丁**，并配置为连接 **当前 NAS 的 IP:4399**。
- 若音箱仍走原厂固件或连到别的服务器，说任何唤醒词都不会有反应，也**不会**听到验证码提示。

## 3. 建议按顺序排查

| 步骤 | 操作 |
|-----|------|
| 1 | 确认音箱已按官方教程打好补丁，并指向 **NAS_IP:4399**（你当前 NAS 的 IP 与 4399 端口）。 |
| 2 | 先试唤醒词 **「你好小智」**（已写在配置里）。若有反应，再试「爸爸最帅」对比。 |
| 3 | 在 NAS 上查日志里的验证码：`docker logs open-xiaoai-xiaozhi`，在输出里搜验证码，再到小智管理后台绑定设备。 |
| 4 | 若确认已连上但仍不灵敏，可在配置里把 `vad.threshold` 再调低（如 `0.03`），保存后重启容器。 |

## 4. 用「纯官方」配置做对比测试

若怀疑与 oxa 增强有关，可临时改用**不依赖 oxa_ext** 的官方格式配置：

1. 在 `docker-compose.yml` 中：
   - 将 config 挂载改为：`./config_simple.py:/app/config.py`
   - 去掉 `./oxa_ext:/app/oxa_ext` 这一行
2. 执行：`docker compose down && docker compose up -d`
3. 再试「爸爸最帅」或「你好小智」。若这样能唤醒，再切回 `config_oxa.py` + `oxa_ext` 继续排查。

## 5. 小结

- **从未听到验证码** 通常说明：要么音箱没连到本机 Open-XiaoAI Server（4399），要么尚未成功触发过一次与小智的对话。
- 配置和关键词生成在 NAS 上已验证正常；重点请先确认 **Rust 补丁 + 连接 NAS:4399** 已正确完成。
